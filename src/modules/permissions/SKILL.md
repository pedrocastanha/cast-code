# Permissions Module

## Overview
Command permission system with danger levels and user approval flow for controlling which agent actions require explicit consent.

## Role in System
Acts as the security gate for dangerous operations. Before the agent executes shell commands, writes files, or performs other risky operations, the permission system determines the danger level and whether user approval is needed. Supports scope-based permissions (per-command, per-session, always-allow).

## Dependencies
- **Depends on**: None (self-contained)
- **Used by**: CoreModule, ToolsModule, TasksModule, REPL
- **External deps**: None

## Key Services/Providers
| Service | Purpose |
|---|---|
| `PermissionService` | Evaluates danger level of operations, checks permission rules, prompts user for approval when needed. Supports session-scoped and persistent rules. |
| `PromptService` | Generates permission request prompts shown to the user when approval is needed. |

## Key Types/Interfaces
| Type | Purpose |
|---|---|
| `DangerLevel` | Enum: safe, moderate, dangerous, critical — determines approval requirements |
| `PermissionResponse` | User's response: approved/denied, scope (once/session/always) |
| `PermissionScope` | Scope of permission grant: 'once', 'session', 'always' |

## Coding Standards & Patterns
- **Danger level classification**: Operations are classified by danger level. Safe ops (read files) auto-approve; dangerous ops (shell commands, writes) require approval.
- **Approval flow**: When approval is needed, the service generates a prompt via `PromptService`, shows it to the user, and waits for response.
- **Scope-based grants**: Users can grant permission for just this execution (`once`), the entire REPL session (`session`), or permanently (`always`).
- **Rule evaluation**: Permissions are evaluated against a ruleset. First matching rule wins. Default deny for unknown dangerous operations.

## Business Rules
- File reads are safe (auto-approved).
- File writes are moderate (may require approval depending on user settings).
- Shell command execution is dangerous (always requires approval unless explicitly allowed).
- Destructive operations (delete, force push) are critical (always require approval).
- Permission grants with `session` scope last until the REPL session ends.
- Permission grants with `always` scope persist across sessions (stored in config or `.cast/`).

## Circular Dependencies
None.

## Working on This Module
- **Adding new danger levels**: Extend the `DangerLevel` type and update the evaluation logic in `PermissionService`.
- **Custom permission rules**: The rule evaluation logic can be extended to support project-specific rules (e.g., allow writes in `src/` but not in `config/`).
- **Prompt customization**: Permission prompts are generated by `PromptService`. Modify to change how approval requests are displayed.
- **Testing**: Test the approval flow with various danger levels and scopes. Ensure `always` grants persist correctly.
- **Integration**: Core and Tools modules call `PermissionService.checkPermission()` before executing operations. The REPL handles the user interaction.
